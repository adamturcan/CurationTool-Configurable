# Medical Mock

A small mock NLP service for medical text. The response shape is on purpose different from SDU, so the existing SDU adapters cannot reach it. The platform has a second set of adapters in `server/src/adapters/*/MockMedical*.ts` that translate this shape into the platform's internal types. We use this to show that pointing the platform at a service with a different response shape needs new adapters, but no changes to the routes, the database or the UI.

## What it does

Different NLP services return data in different shapes. To make the demo realistic, this mock uses a shape that does not match SDU rather than a contrived one. Entities are grouped by type, responses are wrapped in a metadata object, and the field names are different.

## Endpoints

It exposes four POST endpoints and one GET endpoint:

* `POST /v2/clinical-entities` for NER. Entities are grouped by type (`DISEASE`, `MEDICATION`, `ANATOMY`, `PROCEDURE`), spans are `[start, end]` arrays and the score field is called `confidence`.
* `POST /v2/sectionize` for sectioning. Returns `sections` with a `name` and a `char_range: { from, to }`.
* `POST /v2/categorise` for classification. Returns a list of `categories` with optional `subcategories`.
* `POST /v2/translate-en-medical` for abbreviation expansion. The request field is called `input`, not `text`. The expanded text is returned at `output.expanded_text`.
* `GET /v2/languages` returns the supported languages as `{ supported: [{ code, name }] }`.

The matching adapters in `server/src/adapters/*/MockMedical*.ts` translate these into the `NerSpan[]`, `Segment[]`, `ClassificationResult[]` and `TranslationResponse` types the platform expects.

## Sample documents

The texts in `samples/` and the prepared answers were all generated by an AI model. They are not real medical records, contain no real patient data, and the annotations are not authoritative. They only exist so the mock can return plausible looking responses for the demo.

The folder contains three short fictional notes referring to "Patient A", "Patient B" and "Patient C", each with a different condition. If the input does not match a sample, the mock falls back to a generic empty response so the platform still gets valid JSON.

## How to run it

```bash
cd mocks/medical-mock-clinical
npm install
npm start
```

You should see something like `loaded 3 sample document(s)` and `listening on http://localhost:5002`.

There is also a Dockerfile if you prefer:

```bash
docker build -t medical-mock-clinical mocks/medical-mock-clinical
docker run --rm -p 5002:5002 medical-mock-clinical
```

## Pointing the platform at this mock

1. Start the platform with `docker compose up` and this mock with `npm start`.
2. Restart the backend so the `MockMedical*` adapters are picked up at startup.
3. Log in as `admin` and open the Services page.
4. For each of the four service rows, change the URL to the matching `http://localhost:5002/v2/...` path and switch the adapter dropdown to `mock-medical`.
5. Save and open the curation view. Pasting one of the sample texts and running NER should return medical entities normalised into the same span list the platform always uses.

The point of the demo is that we swapped the upstream NLP provider for one with a different response shape, and the only code that changed is the new adapters.
